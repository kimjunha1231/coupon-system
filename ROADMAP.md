# 선착순 쿠폰 시스템 성능 고도화 공동 프로젝트 로드맵

서버 개발 유경험자(리더)와 서버 미경험 팀원의 협업 및 단계적 성장을 고려하여 재설계된 로드맵입니다.
"단순 기능 구현 ➡️ 문제 인식 ➡️ 고도화"의 학습 루프를 기반으로 설계되었습니다.

---

## 📌 Phase 1. 기본 MVP 구현 및 동시성 이슈 마주하기
> **목표:** 서비스의 최소 기능(MVP)을 빠르게 구현하고, 대량 요청 시 발생할 수 있는 데이터 정합성 문제를 코드로 이해합니다.

### 1-1. 단일 사용자 Happy Path 구현 (동시성 고려 X)
- **목표:** 1명의 사용자가 쿠폰을 발급받는 정상적인 흐름을 완성합니다.
- **백엔드 (팀원 주도):**
  - Spring Boot와 Spring Data JPA 기본 세팅
  - User 엔티티 및 Coupon 엔티티 설계 (쿠폰 총 수량 필드 포함)
  - 단순 비즈니스 로직 작성 (`if (coupon.getQuantity() > 0) { coupon.decrease(); save(); }`)
  - 기본적인 CRUD API 작성 (쿠폰 조회, 회원가입 등)
- **프론트엔드 (팀원 주도):**
  - React를 활용하여 쿠폰 목록 및 발급 버튼이 있는 화면 구성
  - axios/fetch를 활용하여 백엔드 API 연동
- **검증:** 로컬 브라우저에서 단일 사용자가 버튼을 클릭해 쿠폰이 정상 발급되는지 손수 테스트

### 1-2. 의도적인 동시성 이슈 유발 및 발견
- **목표:** 동시 요청이 발생했을 때 데이터가 어떻게 깨지는지 확인합니다.
- **백엔드 (리더 주도 + 팀원 참관):**
  - `Junit5`와 `ExecutorService`, `CountDownLatch`를 사용해 동시에 100~1,000개의 쿠폰 발급 요청을 보내는 멀티스레드 테스트 코드 작성
- **분석 & 토론 (공동):**
  - 테스트 실행 후 DB에 저장된 발급 갯수가 설정된 제한 수량보다 더 많이 발급(Race Condition)되는 현상 눈으로 확인
  - 스레드 세이프하지 않은 코드와 트랜잭션 범위 내에서의 동시성 문제 학습
- **프론트엔드 (팀원 주도):**
  - 쿠폰 발급 버튼 중복 클릭 방지 처리 (Debounce/Throttle) 구현 및 필요성 체득

### 1-3. 데이터베이스 락(Lock)을 통한 1차 해결
- **목표:** 데이터베이스 수준에서 정합성을 맞추는 락 메커니즘을 학습하고 적용합니다.
- **백엔드 (공동 - 짝 프로그래밍):**
  - 자바 레벨 해결책 (`synchronized`) 적용 및 다중 서버(WAS) 환경에서의 한계점 이해
  - 데이터베이스 격리 수준(Isolation Level) 학습
  - JPA `@Lock`을 활용한 **비관적 락(Pessimistic Lock)** 및 **낙관적 락(Optimistic Lock)** 구현 및 테스트 통과 확인

## 📌 Phase 2. 안정적인 인프라 구축 및 배포 자동화
> **목표:** 실제 부하 테스트를 진행할 수 있도록 로컬 환경을 벗어나 클라우드 환경에 격리된 인프라를 구축하고 배포를 자동화합니다.

### 2-1. 도커(Docker)를 이용한 애플리케이션 환경 통일
- **목표:** 백엔드 애플리케이션과 DB를 컨테이너화하여 어디서나 동일하게 구동하도록 설정합니다.
- **역할 분담:**
  - **리더:** Dockerfile 및 Docker Compose 설정 파일 작성
  - **팀원:** 로컬에서 Docker를 실행하여 DB(MySQL 등)를 띄우고 애플리케이션 빌드/실행 환경 실습

### 2-2. 클라우드 인프라 구축 및 WAS/DB 분리
- **목표:** 부하 테스트 시 서로 자원을 간섭하지 않도록 인프라를 분리합니다.
- **역할 분담:**
  - **리더:** AWS VPC 설계 및 EC2(WAS용), RDS(DB용) 인프라 구축
  - **팀원:** RDS 연동을 위한 Spring Boot의 `application.yml` 환경 설정 수정 및 로컬 테스트

### 2-3. CI/CD 배포 자동화 파이프라인
- **목표:** 코드 수정 시 자동으로 빌드 및 배포되는 환경을 만듭니다.
- **역할 분담:**
  - **리더:** GitHub Actions 또는 Jenkins를 이용한 배포 스크립(CI/CD) 설계
  - **팀원:** 간단한 코드 수정 후 푸시하여 자동 배포 흐름을 경험하고 로그 확인 방법 학습

## 📌 Phase 3. 부하 테스트를 통한 병목 지점(Bottleneck) 탐색
> **목표:** 시스템에 극한의 트래픽을 발생시켜 어느 구간에서 병목이 생기고 무너지는지 수치와 지표로 확인합니다.

### 3-1. 테스트 시나리오 설계 및 k6 작성
- **목표:** 선착순 이벤트의 특징인 순간적 폭발 트래픽(Spike Traffic) 상황을 코드로 작성합니다.
- **역할 분담:**
  - **리더:** k6 부하 테스트 스크립트 작성 및 시나리오 설계 (예: 가상 유저 1,000명이 동시에 30초 동안 요청)
  - **팀원:** k6 실행 및 기본적인 TPS(초당 처리량) 결과 분석법 학습

### 3-2. Prometheus & Grafana 모니터링 구축
- **목표:** 부하 상황에서의 서버 자원 상태를 시각화합니다.
- **역할 분담:**
  - **리더:** Prometheus, Grafana 컨테이너 설정 및 JVM/DB 커넥션 모니터링 대시보드 구축
  - **팀원:** 성능 테스트 도중 대시보드를 보며 CPU 임계점 도달 및 DB Connection Pool 부족 현상 관찰

### 3-3. 1차 병목 지점 분석
- **공동:** DB Lock으로 인한 Connection Pool 자원 고갈과 Latency(응답 지연) 증가의 상관관계 분석 및 리포트 작성

---

## 📌 Phase 4. 1차 성능 개선: 인메모리 DB를 통한 병목 해소
> **목표:** 가장 무거운 병목 구간인 RDB 조회를 걷어내고, 고성능 메모리 기반 저장소를 활용해 처리량을 대폭 끌어올립니다.

### 4-1. Redis 캐싱 및 원자적 연산 적용
- **목표:** RDB 직접 접근을 줄이고 Redis를 사용해 잔여 수량을 관리합니다.
- **역할 분담:**
  - **리더:** Redis의 `DECR` 원자적(Atomic) 연산을 이용한 쿠폰 수량 차감 알고리즘 설계
  - **팀원:** Spring Data Redis를 활용한 간단한 Redis 연결 설정 및 조회 캐싱 로직 작성

### 4-2. 분산 환경에서의 동시성 제어 (분산 락)
- **목표:** 여러 WAS 대수가 존재할 때의 데이터 정합성을 보장합니다.
- **백엔드 (공동):**
  - Lettuce와 Redisson 라이브러리의 차이점 학습
  - Redisson을 활용한 분산 락(Distributed Lock) 구현 및 1차 락과의 성능 비교

### 4-3. 성능 비교 부하 테스트
- **공동:** k6 테스트를 재실행하여 응답 시간(Latency) 및 TPS 개선 수치 정량적 비교 분석 (예: DB Lock 대비 TPS 5배 향상 확인)

---

## 📌 Phase 5. 2차 성능 개선: 비동기 큐를 이용한 대기열 시스템 (심화)
> **목표:** 시스템이 감당할 수 없는 수준의 트래픽이 와도 서버가 죽지 않고, 들어온 요청을 순서대로 안전하게 처리하는 최종 아키텍처를 완성합니다.

### 5-1. Kafka 메시지 브로커 도입
- **목표:** 발행/구독(Pub/Sub) 모델을 이해하고 비동기 이벤트 처리를 위한 Kafka 인프라를 연동합니다.
- **역할 분담:**
  - **리더:** Kafka 브로커 로컬/클라우드 설정 및 토픽 설계, Producer 구현
  - **팀원:** Consumer 애플리케이션 뼈대 작성 및 간단한 메시지 수신 테스트

### 5-2. 대기열(Queue) 아키텍처 및 비동기 처리 구현
- **목표:** 요청 즉시 대기 번호를 부여하고, 백엔드에서 감당할 수 있는 속도로 쿠폰을 발급해 DB에 저장하는 파이프라인을 완성합니다.
- **역할 분담:**
  - **리더:** 비동기 발급 로직 처리 및 DB 저장(Consume) 구현
  - **팀원:** React 화면에서 대기 순번을 폴링(Polling)하거나 SSE(Server-Sent Events)를 통해 실시간으로 진행률을 표기하는 프론트 화면 고도화

---

## 📌 Phase 6. Spring AI + Vector DB를 활용한 RAG 기반 맞춤형 데이터 파이프라인 구축 (심화)
> **목표:** 대량의 선착순 실패 트래픽을 비동기로 처리하며, 유저 맞춤형 혜택을 실시간으로 추천하는 지능형 RAG 아키텍처를 완성합니다.

### 6-1. 데이터 임베딩 및 Vector DB 설정
- **목표:** 상품 및 쿠폰 도메인 데이터를 벡터 데이터로 변환하고 Vector DB에 인덱싱합니다.
- **역할 분담:**
  - **리더:** Spring AI 설정 및 pgvector 또는 Redis Vector Search 연동
  - **팀원:** 추천 시스템에 활용할 기본 원본 데이터(텍스트) 준비 및 정제 작업

### 6-2. 비동기 RAG 파이프라인 구현
- **목표:** 메인 선착순 트래픽을 방해하지 않도록 Kafka Consumer가 무거운 AI 연산을 처리하도록 아키텍처를 격리합니다.
- **역할 분담:**
  - **리더:** RAG Retrieval 단계 구현 (유사도 검색 기반 최적 컨텍스트 추출 및 LLM 프롬프트 전송)
  - **팀원:** AI 추천 결과 포맷팅 및 프론트 혜택 추천 팝업 렌더링

### 6-3. 부하 테스트 및 비용 최적화
- **공동:** AI 파이프라인 가동 시의 Latency 측정 및 최적의 Throughput 도출, API 사용 비용 최소화를 위한 모델 비교 테스트
